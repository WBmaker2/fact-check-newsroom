# 팩트체크 편집국 추가 기능·디자인·UI 개선 분석

## 1. 문서 목적

- 작성일: 2026-07-15
- 분석 대상: [GitHub Pages 공개 앱](https://wbmaker2.github.io/fact-check-newsroom/)
- 기준 화면: 데스크톱 1440×900, 모바일 375×812
- 대상 사용자: 초등 4~6학년 학생과 수업을 진행하는 교사
- 범위: 기능, 학습 흐름, 화면 구조, 반응형 UI, 접근성, 시각 완성도

이번 분석은 2026-07-14에 완료한 초등 난이도 개선을 다시 제안하는 문서가 아니다. 쉬운 판정 이름, 자료 3개, 확인할 말 2개, 판정 이유 1개, 핵심 비교 문장 등 이미 적용된 개선은 유지하고, 실제 배포본을 처음부터 4단계까지 직접 진행하며 남아 있는 사용성 문제와 다음 개선 후보를 찾았다.

## 2. 한눈에 보는 결론

현재 앱은 편집국 콘셉트와 교육용 가상 자료라는 경계가 명확하고, 시각 디자인도 일반적인 학습 카드형 웹앱보다 정체성이 좋다. 초등학생용 문구도 이전보다 크게 쉬워졌다.

그러나 체감 난이도를 높이는 가장 큰 원인은 내용보다 화면 이동 방식과 반복 조작이다.

1. **단계가 바뀌어도 이전 스크롤 위치가 남는다.** 3단계에서 4단계로 이동한 직후 `scrollY`가 760px인 상태로 시작해, 4단계 제목·주장·핵심 비교·해결 순서가 화면 위로 사라졌다. 학생은 판정 선택지만 갑자기 보게 되어 문제를 더 어렵게 느낄 수 있다.
2. **국어 사건 하나를 끝내는 최소 클릭이 약 43회다.** 이 중 자료 살펴보기의 단서 열기만 16회이고, 자료 비교는 8회다. 읽기 난이도는 낮아졌지만 반복 조작 부담은 여전히 크다.
3. **모바일 단계 표시가 가로로 잘린다.** 375px 화면에서 1~3단계와 4단계 일부만 보이고, 5단계는 화면 밖에 있다. 가로로 밀 수 있다는 표시도 없다.
4. **이전 단계로 돌아갈 방법이 없다.** 현재의 `처음부터`는 전체 초기화이므로, 실수한 학생이 직전 자료를 다시 확인하기 어렵다.

따라서 다음 개선 순서는 `단계 전환 위치 복구 → 반복 클릭 감축 → 이전 단계 이동 → 모바일 진행 표시 → 교사용 기능과 결과 활용`이 적절하다.

## 4. 우선순위별 개선안

### P0. 단계 전환 시 제목 위치와 키보드 초점 복구

#### 확인된 문제

- 2단계 진입 직후 `scrollY: 48px`로 시작해 진행 표시 상단이 일부 가려졌다.
- 긴 2단계의 아래쪽에서 3단계로 이동하자 `scrollY: 1115px`로 유지되어 3단계 제목과 앞부분 자료가 보이지 않았다.
- 3단계에서 4단계로 이동하자 `scrollY: 760px`로 유지되어 다음 내용이 화면 위로 사라졌다.
  - `이 주장은 맞을까요?`
  - 확인할 주장
  - 핵심 비교
  - 세 단계 해결 순서
- 이 현상은 사용자가 어렵다고 느낀 4단계에서 특히 치명적이다. 화면에 설명은 존재하지만 실제 진입 시 보이지 않을 수 있기 때문이다.

#### 권장 변경

- `stage`가 바뀔 때 `<main>` 또는 현재 단계의 `<h1>`을 화면 위로 이동한다.
- 단순히 `window.scrollTo(0, 0)`만 호출하지 말고, `<main tabIndex="-1">`에 초점을 보내 새 단계가 시작되었음을 키보드·스크린리더 사용자에게도 알린다.
- 고정 헤더 높이를 고려해 `scroll-margin-top`을 적용한다.
- 모션 감소 설정에서는 부드러운 스크롤 대신 즉시 이동한다.

#### 수용 기준

- 모든 `다음` 버튼 클릭 후 새 단계의 제목이 첫 화면 안에 보인다.
- 320×568, 375×812, 768×1024, 1440×900에서 단계 진입 후 `scrollY`가 제목을 가리지 않는 범위다.
- 새 단계의 제목 또는 메인 영역에 프로그램 초점이 이동한다.
- 브라우저 뒤로가기나 모달 초점 복귀 동작과 충돌하지 않는다.

#### 예상 작업량

- Codex 기준: 30~60분
- 수동 구현 기준: 1~2시간

### P1. 한 사건의 반복 클릭을 약 43회에서 25회 안팎으로 감축

#### 현재 최소 동작 수

| 구간 | 최소 클릭 수 |
|---|---:|
| 시작·사건 선택 | 2 |
| 확인할 말 2개 선택·이동 | 3 |
| 첫 자료 3개의 단서 4개씩 열기·이동 | 13 |
| 자료 3개 × 확인할 말 2개 비교·근거 선택·이동 | 8 |
| 첫 판단 | 3 |
| 새 자료 단서 열기·이동 | 5 |
| 새 자료 비교·근거 선택·이동 | 4 |
| 마지막 판단 | 3 |
| 제목 선택·결과 보기 | 2 |
| **합계** | **43** |

버튼 크기와 표현이 쉬워도 같은 패턴을 여러 번 반복하면 집중력이 먼저 떨어진다. 특히 2단계에서 모든 자료의 네 단서를 반드시 하나씩 열게 하는 방식은 학생이 내용을 읽기보다 버튼 완료에 집중하게 만들 수 있다.

#### 권장 변경

1. 첫 자료는 현재처럼 네 질문을 하나씩 열어 출처 확인 방법을 연습한다.
2. 두 번째·세 번째 자료에는 `단서 한눈에 보기` 버튼 하나를 제공한다.
3. 새 자료 단계는 이미 배운 방식의 복습이므로 네 번 누르게 하지 않고 한 번에 펼친다.
4. 3단계에서는 먼저 `도움 되는 자료인가요?`를 고르게 한 뒤, 도움 된 자료만 확인할 말과 비교하게 한다.
5. 선택하지 않은 자료에는 정답 판정 후 짧게 `왜 근거로 쓰기 어려운지`를 보여 준다.

이 방식은 출처 확인과 근거 비교라는 학습 목표는 유지하면서 의미 없는 완료 클릭을 줄인다.

#### 수용 기준

- 국어 대표 흐름의 최소 클릭이 25회 안팎이다.
- 첫 자료에서는 `누가·언제·어떻게·누구/어디` 네 단서를 계속 연습한다.
- 학생이 관련 없는 자료를 무조건 건너뛰지 않고, 근거로 쓰기 어려운 이유를 한 번은 확인한다.
- 네 교과 사건의 정답 판정 규칙은 바뀌지 않는다.

#### 예상 작업량

- Codex 기준: 3~5시간
- 수동 구현 기준: 1~2일

### P1. 이전 단계로 돌아가 자료를 다시 볼 수 있게 하기

#### 확인된 문제

- 현재 헤더에는 `업데이트 내역`과 `처음부터`만 있다.
- 판정 중 원문을 다시 보고 싶어도 직전 단계로 돌아갈 수 없다.
- `처음부터`는 전체 상태를 지우므로 실수 복구 수단으로 쓰기 어렵다.

#### 권장 변경

- 각 단계 아래 행동 영역에 `이전 단계` 보조 버튼을 둔다.
- 뒤로 이동해도 이미 고른 답과 펼친 단서는 유지한다.
- 판정 단계에는 더 직접적인 `자료 다시 보기` 링크를 추가해 3단계의 선택 상태를 유지한 채 이동한다.
- 첫 판단 이후 새 자료 단계에서는 과거 판단을 수정하는 기능과 단순히 자료를 다시 보는 기능을 구분한다.

#### 수용 기준

- `처음부터`를 누르지 않고 직전 단계로 이동할 수 있다.
- 뒤로 갔다가 다시 와도 선택 상태가 보존된다.
- 최종 결과에 저장되는 첫 판단과 마지막 판단의 시점은 바뀌지 않는다.
- 키보드 탭 순서에서 `이전`이 `다음`보다 먼저 온다.

#### 예상 작업량

- Codex 기준: 2~4시간
- 수동 구현 기준: 0.5~1일

### P1. 모바일 진행 표시를 한 화면에 보이도록 재설계

#### 확인된 문제

- 375px 화면에서 5단계 진행 표시가 `min-width: 520px`로 렌더링된다.
- 1~3단계와 4단계 일부만 보이고 5단계는 화면 밖에 있다.
- 문서 전체에는 가로 스크롤이 없어서 기존 테스트는 통과하지만, 진행 표시 내부에는 별도 가로 스크롤이 생긴다.
- 모바일 헤더의 두 버튼이 두 번째 줄을 차지해 헤더와 진행 표시만으로 약 200px 이상의 세로 공간을 사용한다.

#### 권장 변경

- 모바일에서는 `2/5 자료 보기` 한 줄과 진행 막대를 사용한다.
- 완료·현재·남은 단계를 모두 글자로 나열하지 말고 현재 단계와 전체 수만 표시한다.
- `업데이트 내역`은 아이콘 보조 메뉴로 줄이고, `처음부터`는 위험도가 있는 초기화 동작이므로 시각적으로 과도하게 강조하지 않는다.
- 데스크톱의 5단계 표시 방식은 유지한다.

#### 수용 기준

- 320px에서도 진행 표시 내부 가로 스크롤이 없다.
- 현재 단계 이름과 `현재/전체` 숫자가 한 화면에 보인다.
- 헤더와 진행 표시를 합친 높이가 375×812 화면의 25%를 넘지 않는다.
- 화면 낭독기에는 다섯 단계 전체와 현재 단계 정보가 계속 제공된다.

#### 예상 작업량

- Codex 기준: 1~2시간
- 수동 구현 기준: 2~4시간

### P1. 다음 행동 버튼을 짧은 화면에서도 찾기 쉽게 하기

#### 확인된 문제

- 1440×900의 1단계에서도 선택을 마친 뒤 `자료 살펴보기` 버튼이 첫 화면 아래에 있다.
- 3·4단계는 세로 길이가 길어 완료 상태와 다음 버튼을 찾기 위해 크게 이동해야 한다.
- 단계 이동 후 스크롤 위치 문제와 결합하면 학생이 어디서 무엇을 해야 하는지 더 혼란스러워진다.

#### 권장 변경

- 모바일과 높이가 짧은 화면에서 하단 행동 영역을 `position: sticky`로 제공한다.
- 진행 수치와 다음 버튼만 남긴 얇은 바 형태로 만들고, 본문을 가리지 않도록 안전 영역과 여백을 둔다.
- 비활성 버튼 옆에는 `무엇을 더 해야 하는지`를 짧게 표시한다.

#### 수용 기준

- 768px 이하 높이에서도 현재 완료 수와 다음 행동이 같은 화면에 있다.
- 고정 행동 바가 마지막 문장이나 선택지를 가리지 않는다.
- 확대 200%에서도 버튼과 상태 문구가 겹치지 않는다.

#### 예상 작업량

- Codex 기준: 1~2시간
- 수동 구현 기준: 2~4시간

### P2. 판정 후 학습 피드백을 한 문장 더 제공

#### 권장 변경

- 맞힌 뒤 곧바로 다음 단계로 넘기기 전에 다음 두 문장을 짧게 보여 준다.
  - `왜 맞는지`: 원문에서 장소와 날짜가 모두 같기 때문
  - `왜 다른 선택지는 아닌지`: 일부만 맞음은 범위가 다를 때, 아직 모름은 자료가 없을 때 사용
- 4단계의 네 판정 차이는 첫 사건에서만 자세히 설명하고, 이후 사건에서는 접을 수 있게 한다.

#### 기대 효과

- 정답 맞히기에서 개념 구분 학습으로 이어진다.
- `아니에요`와 `아직 몰라요`를 구분하는 학습 목표가 더 선명해진다.

#### 예상 작업량

- Codex 기준: 1~2시간
- 수동 구현 기준: 2~4시간

### P2. 교사용 빠른 진행 모드와 인쇄용 안내 추가

#### 권장 기능

- 시작 화면에 작은 `교사용 수업 안내` 링크를 추가한다.
- 새 창이나 모달에 다음 정보를 제공한다.
  - 권장 학년과 시간
  - 단계별 발문 예시
  - 정답을 바로 말하지 않는 힌트
  - 네 판정의 차이
  - 사건별 핵심 학습 목표
- `한 장으로 인쇄` 스타일을 제공한다.
- 학생용 화면에는 정답표를 노출하지 않는다.

#### 기대 효과

- 교사가 앱을 미리 끝까지 풀어 보지 않아도 수업을 시작할 수 있다.
- 대형 화면 시연, 모둠 활동, 개별 활동에 같은 앱을 활용하기 쉬워진다.

#### 예상 작업량

- Codex 기준: 2~4시간
- 수동 구현 기준: 0.5~1일

### P2. 결과 카드 인쇄·이미지 저장 기능 검토

#### 권장 변경

- 서버 전송 없이 브라우저 안에서만 `결과 카드 인쇄` 기능을 제공한다.
- 가능하면 PNG 저장을 추가하되, 외부 라이브러리 용량이 커지면 인쇄 기능을 우선한다.
- 이름이나 학급을 입력받지 않아 현재의 개인정보 비수집 원칙을 유지한다.

#### 주의점

- 게시·공유 버튼은 추가하지 않는다.
- 학생 결과를 서버나 브라우저 저장소에 자동 저장하지 않는다.
- 결과에 `교육용 가상 사건` 표시를 유지한다.

#### 예상 작업량

- 인쇄만: Codex 기준 1~2시간
- PNG 포함: Codex 기준 3~5시간

### P2. 사건 선택 화면에 추천과 학습 목표 표시

#### 권장 변경

- 첫 사용자를 위해 국어 사건에 `처음이라면 추천` 표기를 추가한다.
- 사건마다 한 줄 학습 목표를 보여 준다.
  - 국어·매체: 원문과 옮긴 글 구분
  - 수학·통계: 일부 조사와 전체의 차이
  - 과학·자료: 숫자 두 개 직접 비교
  - 사회·공공자료: 처음 자료와 수정 자료 구분
- `가상 자료`, `약 10~15분`처럼 모든 행에 반복되는 정보는 사건 목록 위 공통 안내로 옮긴다.

#### 기대 효과

- 선택을 어려워하는 학생이 빠르게 시작할 수 있다.
- 빈 가로 공간을 학습 정보로 활용하면서 목록 높이는 줄일 수 있다.

#### 예상 작업량

- Codex 기준: 1시간 내외
- 수동 구현 기준: 1~2시간

### P3. 세부 시각·완성도 개선

1. **favicon 404 수정**
   - 공개 사이트가 `https://wbmaker2.github.io/favicon.ico`를 요청해 404가 발생한다.
   - 프로젝트 경로를 고려한 favicon 링크와 실제 아이콘 파일을 추가한다.
2. **의견 조각 상태 개선**
   - `좋은 소식 · 의견`이 비활성 버튼처럼 보여 고장 난 선택지로 오해할 수 있다.
   - `판정에서 빼기 완료` 또는 체크·잠금 표시를 사용해 학습상 제외된 상태임을 강조한다.
3. **긴 단계의 구획 강화**
   - 3단계 자료 사이 구분은 좋지만, 현재 자료 번호인 `자료 1/3`을 추가하면 긴 화면에서 위치를 잃지 않는다.
4. **헤더 초기화 동작의 위험도 표현**
   - `처음부터`는 전체 진행을 지우는 동작이므로 업데이트 버튼과 같은 무게로 상시 강조할 필요가 없다.
   - 진행 중에는 보조 스타일을 유지하고 확인 모달에 `현재 선택이 모두 사라짐`을 명확히 표시한다.
5. **디자인 토큰 문서화**
   - 현재 색·간격·폰트 토큰이 잘 분리되어 있다. `docs/design-system.md`에 사용 원칙과 모바일 변형을 기록하면 이후 교과 팩 확장 시 일관성을 지키기 쉽다.

## 5. 추천 구현 순서

### 1차: 즉시 수정

- 단계 전환 스크롤·초점 복구
- 해당 회귀 E2E 추가
- favicon 404 수정

**완료 기준:** 새 단계의 제목이 항상 보이고, 공개 사이트 콘솔 오류가 없다.

### 2차: 학습 부담 줄이기

- 단서 열기 반복 축소
- 자료 비교 조작 축소
- 이전 단계와 `자료 다시 보기` 추가
- 대표 흐름 최소 클릭 수 테스트 또는 기록

**완료 기준:** 한 사건 최소 클릭 약 25회, 핵심 출처·근거 학습 목표 유지.

### 3차: 모바일 UI

- 모바일 진행 막대
- 헤더 높이 축소
- 짧은 화면용 고정 행동 바

**완료 기준:** 320px에서도 진행 표시가 잘리지 않고 다음 행동을 쉽게 찾을 수 있다.

### 4차: 수업 활용 확장

- 교사용 수업 안내와 인쇄
- 결과 카드 인쇄
- 사건 추천과 학습 목표
- 판정 후 개념 피드백

**완료 기준:** 교사가 별도 설명 자료 없이 10~15분 수업을 진행할 수 있다.

## 6. 테스트 보강 제안

기존 테스트는 가로 스크롤, 44px 터치 영역, 모달 초점, 전체 사건 완료를 잘 확인한다. 다음 회귀 테스트를 추가하면 이번 분석에서 발견한 문제를 막을 수 있다.

1. 단계 전환 후 현재 단계 제목이 뷰포트 안에 있는지 확인
2. 단계 전환 후 `<main>` 또는 `<h1>`에 초점이 있는지 확인
3. 모바일 진행 표시 자체의 `scrollWidth <= clientWidth` 확인
4. 320×568에서 고정 행동 바가 마지막 선택지를 가리지 않는지 확인
5. `이전 단계 → 다시 다음 단계`에서 선택값이 보존되는지 확인
6. 대표 국어 사건의 최소 조작 수가 정한 상한을 넘지 않는지 확인
7. 공개 배포본에서 콘솔 오류와 404 요청이 없는지 확인

## 7. 이번 개선에서 하지 않는 것이 좋은 것

- 실제 뉴스 URL을 입력받아 자동으로 진위를 판정하는 기능
- AI가 근거 없이 정답을 생성하는 기능
- 로그인, 학생 계정, 서버 저장, 순위표
- 정답률 점수와 경쟁 중심의 과도한 게임화
- 모든 화면을 색이 다른 카드와 배지로 채우는 시각 변경
- 제목 자유 입력을 기본값으로 바꾸는 것

이 기능들은 현재 앱의 장점인 안전한 합성 자료, 개인정보 비수집, 교과 검수, 편집국다운 절제된 시각 언어를 약화할 가능성이 크다.

## 9. 최종 제안

가장 먼저 고쳐야 할 것은 4단계 내용을 더 단순하게 바꾸는 일이 아니라, **학생이 4단계의 설명을 실제로 볼 수 있게 단계 전환 위치를 복구하는 일**이다. 그다음 반복 클릭을 줄이고 이전 단계 이동을 제공하면, 이미 적용된 쉬운 문구와 핵심 비교 안내가 제대로 효과를 낼 수 있다.

현재 디자인 방향은 유지하는 것이 좋다. 다음 개선은 색상이나 장식을 늘리기보다 `보이는 순서`, `누르는 횟수`, `실수 복구`, `모바일 공간`, `교사의 수업 준비`를 다듬는 데 집중하는 편이 교육 효과가 크다.

## 10. 구현 기록

- 2026-07-15: 사용자가 이 문서의 우선순위 순서대로 구현을 요청했다.
- P0: 단계 전환 시 화면을 위로 이동하고 메인 영역에 초점을 보내도록 변경했다. GitHub Pages 하위 경로를 따르는 favicon도 추가했다.
- P1: 자료별 네 단서를 한 번에 펼치고, 도움 되는 자료를 먼저 고른 뒤 선택한 자료만 비교하도록 바꿨다. 대표 흐름의 최소 조작은 약 43회에서 약 27회로 줄었다.
- P1: 새 자료 보기와 새 자료 비교를 별도 상태로 나누고, 이전 단계로 돌아가도 확인한 단서·비교·판정이 유지되도록 했다.
- P1: 모바일 진행 표시를 현재 단계와 진행 막대로 바꾸고, 모바일과 짧은 화면에 다음 행동 영역을 고정했다.
- P2: 판정 뒤 왜 맞는지 알려 주는 학습 피드백, 교사용 수업 안내와 인쇄, 결과 카드 인쇄, 사건 추천과 학습 목표를 추가했다.
- P3: 의견 조각의 제외 완료 상태, 자료 번호, 초기화 버튼의 보조 표현, 2026-07-15 업데이트 내역을 반영하고 `docs/design-system.md`를 추가했다.

### 검증 기록

- `npm run check`: 린트, 타입 검사, 프로덕션 빌드와 500줄 제한 통과. 최종 단위·컴포넌트 테스트는 45개 통과
- `npm run test:e2e`: 26개 E2E 통과
- 네 교과 사건 전체 완료, 단계 전환 후 `scrollY: 0`, `<main>` 초점 이동, 이전 단계 선택 보존을 확인했다.
- 320px 모바일 진행 표시 내부 가로 스크롤 없음과 44px 이상 터치 영역을 확인했다.
- 실제 375×812 브라우저에서 모바일 진행 막대, 의견 제외 상태, 고정 행동 영역, 자료 번호와 단서 펼치기를 확인했다.
- GitHub Actions 환경 변수로 빌드해 favicon과 정적 자산이 `/fact-check-newsroom/` 하위 경로를 사용하는 것을 확인했다.
